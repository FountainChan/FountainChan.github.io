倪海厦中医 AI 模型：一个开源项目把 2452 页中医经典蒸馏成 Agent Skill。

2452 页讲义 · 849 个医案 · 129 条伤寒论条文 — 蒸馏成一个 3MB 的 AI Skill。当中医遇到大模型，这件事比你想的更硬核。

SECTION 01：当程序员遇到中医

有个程序员朋友，代码写了十年，颈椎废了、脾胃也废了。去三甲医院，医生开了两盒中成药，他回来翻了翻说明书——“功能主治”四个字，连成分都没写清楚。

他问我：中医到底有没有标准？还是全靠老中医的”感觉”？

我说：你问对人了。倪海厦的中医体系，可能是最适合程序员理解的中医流派。因为倪海厦的经方体系是 规则驱动的。

六经辨证不是玄学，是一套完整的 状态机：

太阳（表）→ 阳明（里热）→ 少阳（半表半里）
↓ 失治误治
太阴（脾寒）→ 少阴（心肾阳虚）→ 厥阴（阴阳逆乱）

这就是六经传变规律——像一个有限状态机，每个状态有明确的输入条件（症状）、转移条件（误治/失治）和输出（方剂）。如果你懂状态机，你就懂六经辨证。

现在有人把这个规则系统蒸馏成了一个 AI Agent Skill。

SECTION 02：这个项目做了什么

项目叫 nihaixia，本质上是把倪海厦全部教学资料蒸馏成一个 3MB 的 SKILL.md 文件（35,285 行）。

数据来源：

  • 伤寒论 209 页，277K 字符，129 条条文全覆盖
  • 金匮 419 页，626K 字符，23 篇完整蒸馏
  • 黄帝内经 461 页，227K 字符，71 篇完整蒸馏
  • 神农本草经 339 页，843K 字符，OCR 校正
  • 医案集 849 个医案，已蒸馏 311+ 例
  • 合计 2,452 页，3.5M 字符

这不是简单的 PDF 转文字。项目做了几件技术活：OCR 校正 13 处生僻缺字、全量逐条提取而非摘抄金句、按倪海厦思维框架重组结构、把模糊问诊变成可激活的诊断路径。

SECTION 03：为什么程序员应该关注这个

经方体系天然适合结构化表达。 他的诊断有明确的 if-then 规则：

  • if 脚冷至膝 && 手冷至肘 → 生附子
  • if 汗出恶风 && 卫气不固 → 炮附子
  • if 水肿 && 关节肿痛 → 生硫磺

AI 蒸馏中医是极好的 NLP 工程案例。 2452 页 PDF → 3.5M 字符 → 35,285 行 Skill → 一个可调用的 AI Agent。这个链路对做知识库、RAG 系统的程序员来说，是一个完整的参考实现。

辨证思维和系统设计思维异曲同工。 倪海厦的六健康标准——一觉到天亮、胃口正常、晨起大便、小便 5-7 次、头面冷手足温、晨起阳反应——这不是养生建议，这是 健康状态的监控指标。和系统监控的黄金信号（CPU、内存、IO 延迟）是一个思路。

用少量关键指标判断系统是否健康，中医的辨证思维 × 程序员的系统思维。

SECTION 04：项目的能力边界

当前蒸馏覆盖了 11 大模块：伤寒论 129 条全覆盖、金匮要略 23 篇、黄帝内经 71 篇、针灸教程 216 页、神农本草经 339 页、医案集 311+ 例、天纪体系 286 页、人纪班闭门课、梁冬对话精华、汉唐文章 10 篇、诊疗日志 8 则。

直接激活词：倪海厦、海厦视角、倪师、经方思维、倪海厦会怎么看。

SECTION 05：程序员视角的中医学习路径

如果你是一个想了解中医的程序员，倪海厦可能是最好的入口：

  1. 从六经辨证开始 — 状态机思维，半小时理解框架
  2. 理解经方思维 — 为什么麻黄汤是固定组合而非随意加减
  3. 学十问 — 诊断问诊框架，和系统排查是一个思路
  4. 看医案 — 849 个案例，每个都是一个完整的 debug 日志

这个项目把最后一步的门槛降到了最低——你不需要读 2452 页 PDF，直接问 AI 就行。

SECTION 06：问题与思考

  • 3MB 的 SKILL.md 作为单文件 Prompt 非常大，不是所有模型都能完整加载
  • 蒸馏质量 依赖原始 PDF 质量，部分 OCR 校正可能还有遗漏
  • 中医争议 — 倪海厦经方派观点激进，你需要有自己的判断
  • 不能替代医生 — 项目声明写得很清楚，仅供学习

但从工程角度看，这是一个高质量的 NLP 蒸馏实践。它证明了：传统知识体系如果结构清晰，完全可以蒸馏为 AI 可用的知识库。

SUMMARY

nihaixia 项目做了一个大胆的尝试：把一门传统医学的完整知识体系，蒸馏成一个 AI 可以调用的 Skill。2452 页讲义、849 个医案、129 条伤寒论条文——浓缩成 3MB 的规则引擎。

对于程序员来说，这不仅是学中医的入口，更是一个知识蒸馏的工程参考。
